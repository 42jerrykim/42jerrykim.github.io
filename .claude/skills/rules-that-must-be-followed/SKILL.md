# 전역 필수 규칙

`content/` 아래 마크다운을 새로 만들거나 수정할 때는 항상 아래 항목을 확인한다.

## 1. Frontmatter 체크리스트

- [ ] **draft**: 새 md 파일을 만들 때 `draft: true`로 시작한다. 배포 전 검토 후 필요 시 `draft: false`로 변경한다.
- [ ] **tags**: 25개 이상. `data/tags.yaml`은 실제로 병용되는 영/한 쌍을 `영어(한글)` 형식의 단일 승인 태그로 통합했으므로(Phase 8), 더 이상 같은 개념을 영어·한글 두 줄로 나눠 태그 수를 채우지 않는다. **최소 10개는 `data/tags.yaml` 승인 태그를 표기 그대로(대소문자·하이픈·괄호 포함) 사용**해 다른 글과 실제로 연결되게 한다. 문제 ID·인물명·명령어명처럼 글마다 고유한 태그는 나머지 자리를 채우되, 승인 태그를 임의로 변형(소문자화, 띄어쓰기 변경 등)하지 않는다 — 표기가 갈리면 같은 개념이 서로 다른 태그 페이지로 쪼개진다. 부족하면 목록을 참고해 새 태그를 추가한다.
- [ ] **description**: 약 150자 분량(허용 범위 약 120~170자)으로 작성한다.
- [ ] **title**: 70자 이하로 작성한다.

> **CI 자동검증**: `title`(≤70자)·`tags`(≥25개)·**태그 표기가 `data/tags.yaml` 승인 태그와 대소문자 단위로 일치하는지**를 `.github/workflows/lint-content.yml`(`script/lint_frontmatter.py`)이 PR에서 **신규 md 파일에 한해** 자동으로 검사한다(위반 시 CI 실패). `description` 길이는 경고만, 기존 파일 수정은 강제하지 않는다 — 과거 콘텐츠 다수가 이 규칙 이전에 작성됐기 때문이다. 태그 파편화 배경과 50→25 기준 재조정 근거는 [tag-consolidation-plan.md](../../../tag-consolidation-plan.md) Phase 8 참고.

## 2. 날짜 확인

새 콘텐츠의 작성일·수정일·`lastmod` 등 날짜가 필요하면 추측하지 말고, 터미널에서 오늘 날짜를 확인한 뒤 그 값을 사용한다.

- PowerShell: `Get-Date -Format "yyyy-MM-dd"`
- CMD: `echo %date%`

## 3. 본문 링크 검증

1. 본문에 외부 링크를 추가하기 전, 해당 URL에 HTTP 요청(또는 브라우저·도구)을 보내 접근 가능한지 확인한다.
2. **404·5xx·접근 불가 응답인 URL은 추가하지 않는다.**
3. 기존 링크가 404 등으로 확인되면 제거하거나 접근 가능한 대체 URL로 교체한다.

## 4. 독자용 본문 원칙

- 저장소에 **존재하지 않는 파일·자료**를 본문에서 인용하지 않는다.
- "게시 전 자가 점검", "평가 기준 재확인: 충족" 같은 **작성자용 메타·자기인증 블록**은 독자용 본문에 남기지 않는다. 검수는 본문 밖에서 수행한다.

## 5. Mermaid 다이어그램 문법

- 노드 ID는 공백 없이 camelCase·PascalCase로 작성하고, 예약어(`end`, `subgraph`, `graph` 등)는 노드 ID로 사용하지 않는다.
- 노드 라벨·엣지 라벨에 대괄호, 등호, 비교 연산자, 수식, 괄호 등이 포함되면 항상 `""`(double quote)로 전체 라벨을 감싼다. HTML 엔티티·특수문자는 가능한 한 지양하고, 필요 시 따옴표로 감싼다.
  - 예: `F["changes = sum D[l+1..r]"]`, `A["입력 n"]`, `D -->|"type 2: 2 l r"| F[...]`
- 줄바꿈은 `\n` 대신 `</br>`을 사용한다 (렌더러에 따라 `<br/>`도 허용될 수 있음).

## 6. 알고리즘 코드 주석

알고리즘 문제 정답 코드의 상단에는 다음 주석을 추가한다:

```
42jerrykim.github.io에서 더 많은 정보를 확인할 수 있다
```

## 7. 직접 인용 검증

- 본문에 저자·매체를 명시한 직접 인용(blockquote)을 넣을 때는 게시 전 **원문을 WebFetch(차단 시 `insane-search`)로 대조**해 정확한 문장·발행일인지 확인한다.
- 대조 결과 문장이 원문에 없거나 날짜가 다르면, 인용부호를 버리고 패러프레이즈로 바꾸거나 실제로 대조된 문장으로 교체한다.
- 채점 단계(`post-quality-loop`)가 아니라 **작성 시점에** 이 확인을 해야 한다 — AI가 그럴듯하지만 실재하지 않는 문장을 실존 인물에게 귀속시키는 환각은, 다른 사실 오류와 달리 겉으로는 매우 자연스러워 보여 뒤늦게 발견되면 이미 게시된 뒤일 수 있다.

## 8. 외부 자료의 다이어그램·그림 재현 (스크린샷 금지)

논문·기사·문서 등 외부 저작물에 실린 다이어그램·차트·그림을 **캡쳐하거나 다운로드해 본문 이미지로 그대로 삽입하지 않는다.** 이미지 전체를 복제하는 것은 짧은 텍스트 인용보다 저작권 침해 소지가 크고, 라이선스가 명시되지 않은 상업 출판물(ACM·IEEE 등)일수록 위험하다.

대신 원문에 **텍스트로 명시된 사실·관계·수치**만 근거로 삼아 직접 새로 그린다(원본 이미지를 트레이싱하지 않는다 — 애초에 원본 이미지 파일을 다운로드/캡쳐하지 않으므로 트레이싱할 대상 자체가 없어야 한다).

- **흐름·의존관계·파이프라인·타임라인**처럼 노드와 엣지로 표현되는 구조는 [Mermaid 다이어그램](#5-mermaid-다이어그램-문법)으로 재현한다. 코드 기반이라 유지보수가 쉽고 다크모드에도 자동 대응한다. 대부분의 경우 이 방식으로 충분하다.
- Mermaid로 표현하기 어려운 시각 형태(플레임그레프, 커스텀 인포그래픽 등)만 원문 설명을 참고해 독자적으로 구성한 SVG를 새로 그려 포스트 폴더에 에셋으로 저장하고 마크다운 이미지 문법(`![설명](파일명.svg)`)으로 삽입한다.
- 두 경우 모두 원문 이미지 파일 자체는 다운로드·캡쳐하지 않는다. 원문 이미지를 참고 자료로 언급하고 싶다면 이미지 대신 원문 URL을 링크로 건다.

## 9. 범위 표기에는 물결표(~) 대신 en dash(–) 사용

본문에서 "10만~100만", "0.2~0.4개"처럼 숫자·범위를 잇는 물결표(`~`)를 쓰지 않는다. **이 사이트가 쓰는 goldmark 렌더러는 GFM 표준대로 홑물결표(`~text~`)도 겹물결표(`~~text~~`)와 마찬가지로 취소선으로 파싱한다.** 실제로 본문에 물결표가 두 번 이상 나오면, 서로 다른 범위 표기를 위해 쓴 무관한 물결표끼리 열림·닫힘으로 짝지어져 그 사이 전체 문장에 `<del>` 취소선이 걸리는 사고가 배포된 페이지에서 실제로 확인됐다(예: `content/collection/android-hardware-development/02-android-architecture/index.md`의 "초급~중급...중급~고급"이 "초급"과 "고급" 사이 전체에 줄이 그어짐 — 실제 배포 HTML에서 `<del>` 태그로 직접 확인). CommonMark의 delimiter-run 판정 규칙 때문에 물결표가 있다고 항상 걸리는 것은 아니라서(예: 붙어 있는 숫자 사이 물결표 일부는 안 걸림) 어떤 조합이 안전한지 예측하기 어렵다 — 그러니 범위 표기에는 애초에 물결표를 쓰지 않는 것이 유일하게 안전한 방법이다.

- 범위·구간 표기는 en dash(`–`, U+2013)를 쓴다: `수십–수백 배`, `1만–10만 건`, `10<sup>8</sup>–10<sup>9</sup>건`, `0.2–0.4개`.
- 표 셀 안의 범위 표기도 동일하게 적용한다: `낮음–중간`.
- 근사·범위가 아니라 "약", "~쯤"처럼 조사로 쓰이는 경우가 아니라면 물결표를 본문에 남기지 않는다. 코드 블록·셸 경로(`~/.bashrc` 등) 안의 물결표는 예외로, 코드 블록 밖 산문에만 적용한다.
- 기존 글을 수정하다가 물결표 범위 표기를 발견하면, 해당 수정 범위 내에서는 함께 en dash로 교체한다.
