**SBO(Small Buffer Optimization)**는 "작은 객체는 내부 버퍼에 직접 저장하고, 큰 객체만 힙에 올린다"는 패턴입니다. 본 챕터에서는 SBO 개념과 **std::function**·**std::any** 등 타입 소거 타입의 내부 구조, 성능·ABI 영향을 정리합니다.

## SBO 패턴과 표준 타입 (역사·배경)

**std::string**의 SSO(챕터 03)와 마찬가지로, **std::function**·**std::any** 등 타입 소거를 쓰는 표준 타입은 구현체마다 **내부 버퍼 크기**를 두어, 그 안에 들어가는 객체는 힙 할당 없이 인라인 저장합니다. 이 한계를 넘으면 힙에 할당해 포인터만 보관합니다. 표준은 "어떤 크기까지 SBO인지"를 규정하지 않으므로, 구현체·ABI 문서를 참고해 "작은 클로저는 할당 없음"을 기대할 수 있고, 큰 클로저는 할당이 발생할 수 있음을 인지해야 합니다.

> "Implementations are encouraged to avoid dynamic allocation for small function objects." — ISO C++ (function). SBO는 권장 사항이며, 한계는 구현체 정의입니다.

## SBO 패턴 개념

**Small Buffer Optimization(SBO)**은 "작은 객체는 **내부 버퍼**에 직접 저장하고, 그보다 큰 객체만 **힙**에 올린다"는 패턴입니다. 타입 소거 타입(여러 구체 타입을 하나의 타입으로 담아야 하는 경우)에서, **크기·정렬 제한** 안에 들어오는 객체는 할당 없이 인라인 저장하고, 그렇지 않으면 힙에 두고 포인터만 보관합니다. 이렇게 하면 **할당 비용**을 줄이고, 작은 객체일 때 **캐시 친화성**을 유지할 수 있습니다. 설계 시 "작은 객체"의 한계(몇 바이트까지 인라인할지)를 정해 두고, 그 한계를 넘는 사용은 힙 비용을 감수하는 것으로 이해하면 됩니다.

## std::function 내부

**std::function**은 호출 가능 객체(함수 포인터, 람다, 함수 객체 등)를 **타입 소거**해 저장합니다. 내부적으로는 **작은 객체**는 객체 자체를 **내부 버퍼(SBO)**에 넣고, **큰 객체**는 힙에 할당해 포인터만 보관합니다.

```cpp
std::function<void()> f_small = []() { };  // 캡처 없음 → 대부분 SBO, 할당 없음
std::function<void()> f_big = [huge = BigObject{}]() { };  // 큰 캡처 → 힙 할당 가능
```

SBO 크기 임계값은 구현체마다 다릅니다(보통 수십 바이트 정도). **호출**은 저장된 객체의 `operator()`를 **간접 호출**(함수 포인터 또는 가상 호출)로 수행합니다. **작은 람다**(캡처가 적음)는 SBO에 들어가 할당이 없고, **큰 람다**(캡처가 많거나 클로저가 큼)는 힙 할당이 발생할 수 있습니다. **이동·복사** 시에는 내부 버퍼 또는 포인터만 복사/이동하므로, 저장된 호출 가능 객체의 크기가 크면 이동이 저렴하고, 복사는 호출 가능 객체의 복사 비용이 따릅니다.

## std::any 내부

**std::any**는 **임의의 타입**의 값을 하나 담습니다. 내부적으로는 **type_id**(타입 식별)와 **저장소**를 갖고, 저장소는 **SBO**이거나 **힙**입니다. **any_cast**로 값을 꺼낼 때는 **타입이 일치하는지** 확인하고, 일치하면 저장된 객체에 접근하며, 일치하지 않으면 예외를 던지거나(nullptr 반환 오버로드) null을 반환합니다. 타입 확인과 저장소 접근 비용이 있으므로, "타입을 모르는 값"을 넘길 때만 사용하고, 타입이 컴파일 타임에 알려지면 **variant**나 **템플릿**이 더 효율적입니다. 구현체별로 SBO 크기가 다르므로, 큰 객체를 any에 넣으면 대부분 힙 할당이 발생합니다.

## 타입 소거와 ABI·성능

**ABI 안정성**: std::function, std::any 같은 타입의 **내부 버퍼 크기**는 구현이 정한 값이므로, 버퍼 크기를 바꾸면 **바이너리 호환**이 깨질 수 있습니다. 그래서 구현체는 한 번 정한 SBO 크기를 쉽게 바꾸지 않습니다. **성능** 측면에서는 **SBO가 성공**하면 할당이 없고 호출만 간접 호출 비용이 있고, **SBO가 실패**하면 힙 할당 + 간접 호출이 추가됩니다. 설계 시 "우리 타입 소거에 넣을 객체는 대부분 이 크기 이하"를 의식하고, 그 한계를 넘는 객체가 많으면 다른 설계(예: variant, 템플릿)를 고려하는 것이 좋습니다.

## 평가 기준 (학습 성과 목표)

- **SBO**의 개념(작은 객체는 내부 버퍼, 큰 객체는 힙)과 타입 소거 타입에서의 역할을 설명할 수 있다.
- **std::function**·**std::any**의 내부 구조(SBO·힙·간접 호출)를 설명하고, 작은 람다/객체가 SBO에 들어가는 조건을 구분할 수 있다.
- SBO 한계를 넘는 사용 시 힙 비용·ABI 영향을 이해하고, 필요 시 variant·템플릿 대안을 선택할 수 있다.

## 판단 기준 (언제 쓸고 언제 피할지)

| 상황 | 권장 | 비권장 |
|------|------|--------|
| 타입 소거 필요·객체 작음 | std::function/std::any (SBO 기대) | 큰 객체만 넣기 |
| 타입이 컴파일 타임에 고정 | variant, 템플릿 | std::any 과다 사용 |
| 콜백이 작은 람다 | std::function (SBO 성공) | 큰 클로저로 SBO 실패 유발 |

**적용 체크리스트**: (1) 타입 소거에 넣을 객체 크기가 SBO 한계 이하인지 의식. (2) 한계를 넘으면 variant/템플릿 검토. (3) ABI 호환 시 버퍼 크기 변경 회피.

## 비판적 시각: 한계와 트레이드오프

- **SBO 크기**는 구현체마다 다르고 ABI로 고정되므로, "이 크기면 SBO"라고 가정하면 이식성에 영향을 줄 수 있다. "작을수록 유리" 정도로만 의식하는 것이 안전하다.
- **std::any**는 타입이 런타임에만 알려질 때 유용하다. 타입이 컴파일 타임에 알려지면 variant나 템플릿이 더 효율적이다.

### 자주 묻는 질문 (FAQ)

**Q: SBO란?**  
A: Small Buffer Optimization. std::function·std::any 등이 작은 객체를 내부 버퍼에 저장해 힙 할당을 피하는 패턴입니다. 구현체별 버퍼 크기가 다릅니다.

**Q: std::function이 언제 힙 할당하나요?**  
A: 저장하는 호출 가능 객체(클로저 등)가 내부 버퍼 크기를 넘으면 힙에 할당합니다. 람다 캡처가 많거나 크면 SBO를 벗어날 수 있습니다(챕터 13).

**Q: 타입 소거와 SBO의 관계는?**  
A: 타입 소거로 하나의 타입(function, any)으로 저장하려면 객체를 어딘가에 두어야 합니다. 작으면 내부 버퍼(SBO), 크면 힙입니다.

### 자주 하는 실수

- **큰 클로저를 std::function에 무조건 저장**: SBO 한도 초과 시 힙 할당; 크기를 줄이거나 할당 비용을 인지합니다.
- **구현체별 SBO 크기 무시**: 이식 시 ABI·크기가 다를 수 있으므로 문서 확인합니다.
- **타입 소거 비용만 강조**: SBO로 작은 객체는 할당 없을 수 있음; 크기 한도만 의식합니다.

### 리팩토링 시 주의

콜백을 std::function으로 통일할 때 클로저 크기가 SBO를 넘지 않게 하거나, 넘으면 할당 비용을 수용합니다. 구현체를 바꿀 때 SBO 한도가 달라질 수 있으므로 벤치마크·할당 프로파일을 확인합니다.

### 최종 정리

(1) SBO = 작은 객체 내부 버퍼·힙 회피. (2) std::function·std::any = 타입 소거·SBO 한도 초과 시 힙. (3) 구현체별·ABI 주의. (4) 다음 장 15 = Parameter Passing 전략.

### 선택 플로우 (SBO·function/any)

1. **std::function·std::any** 사용 시 SBO 한도(구현체별)를 인지한다.
2. **큰 클로저** 저장 시 할당이 발생할 수 있음을 수용하거나, 클로저 크기를 줄인다.
3. **핫패스**에서는 템플릿으로 람다를 받아 할당·간접 호출을 피한다(챕터 13).
4. **이식 시** ABI·SBO 크기 차이를 확인한다.
5. **검증**: 할당 프로파일·벤치마크.
