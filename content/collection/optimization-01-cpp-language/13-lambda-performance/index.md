**람다 성능**은 캡처 방식·클로저 크기·전달 방식(템플릿 vs std::function)에 따라 달라집니다. 본 챕터에서는 **by-value vs by-reference** 캡처 비용, 클로저 객체 크기·인라이닝 가능성, **std::function**과의 비교 및 콜백·알고리즘 전달 시 비용을 정리합니다.

## 람다와 클로저 in C++ (역사·배경)

**람다 표현식**은 C++11에서 도입되었고, C++14에서 일반화 캡처·반환 타입 추론 등이 추가되었습니다. C++20에서는 템플릿 람다·consteval 람다 등이 확장되었습니다. 람다는 **클로저 객체**(캡처된 변수를 담은 익명 타입)를 생성하고, 템플릿으로 전달하면 해당 타입이 알려져 인라인되기 쉽고, **std::function**으로 전달하면 타입 소거로 인해 간접 호출·(가능하면) SBO가 적용됩니다. 본 챕터는 캡처·전달 방식별 비용을 정리합니다.

> "The type of a lambda is a unique, unnamed closure type." — ISO C++. 템플릿으로 이 타입을 받으면 인라인·최적화에 유리합니다.

## 캡처 비용

**by-value** 캡처(`[x, y]`)는 해당 변수를 **클로저 객체 내부에 복사(또는 이동)**합니다. 캡처된 값이 크면 클로저 크기가 커지고, 람다 생성 시점에 복사/이동 비용이 듭니다. **by-reference** 캡처(`[&x, &y]`)는 **포인터/참조 크기만** 클로저에 들어가므로 클로저가 작지만, 람다가 **참조한 객체보다 오래 살면** dangling이 됩니다. 따라서 수명이 람다보다 긴 객체만 참조로 캡처해야 합니다.

**캡처 블록 설계**에서는 **필요한 것만** 캡처하고, 크기가 작고 복사가 저렴하면 값으로, 크기가 크거나 수명이 보장되면 참조로 선택합니다.

```cpp
int a = 1, b = 2;
auto by_val = [a, b]() { return a + b; };   // 클로저에 a, b 복사
auto by_ref = [&a, &b]() { return a + b; }; // 클로저에 참조만
// 템플릿으로 받으면 람다 타입이 알려져 인라인 가능
template<typename F>
void apply(F&& f) { f(); }
// std::function으로 받으면 타입 소거, 간접 호출·SBO
void apply(std::function<void()> f) { f(); }
```

`[=]`, `[&]`는 사용하기 쉽지만 불필요한 캡처까지 포함할 수 있으므로, 핫 경로에서는 명시적으로 나열하는 편이 의도와 비용을 파악하기 좋습니다.

## 클로저 객체

람다 표현식은 **고유한 클로저 타입**을 가지며, 그 크기는 **캡처된 멤버의 합**과 정렬에 의해 결정됩니다. 캡처가 없으면 크기가 매우 작고(1바이트 등), 캡처가 많으면 그만큼 커집니다.

**인라인 가능성**: 람다를 **템플릿 파라미터**로 받는 함수(예: `std::sort`의 비교자, `std::for_each`의 함수 객체)는 **호출 시점에 람다의 타입이 알려지므로** 컴파일러가 람다 본문을 인라인할 수 있습니다. 반대로 **std::function**에 람다를 담으면 **타입이 소거**되어, 호출이 **간접 호출**(함수 포인터 또는 가상 호출)로 이루어질 수 있고, 클로저가 크면 **힙 할당**(SBO 실패 시)도 발생할 수 있습니다. 따라서 **콜백·알고리즘에 람다를 넘길 때**는 템플릿으로 받는 API를 쓰는 것이 성능에 유리합니다.

## std::function과의 비교

**std::function**은 임의의 호출 가능 객체를 타입 소거해 하나의 타입으로 담습니다. 내부적으로는 (작은 객체면 SBO, 크면 힙에) 저장하고, 호출 시 **간접 호출**을 합니다. 그래서 **힙 할당**이 발생할 수 있고, **인라인**이 어렵습니다. **람다를 템플릿으로 받는** API(예: `template &lt;typename F&gt; void run(F&& f)`)는 **타입이 그대로 노출**되므로 인라인과 최적화가 잘 됩니다. 콜백·알고리즘 전달 시에는 **템플릿 파라미터로 람다를 받는** 인터페이스를 우선 사용하는 것이 좋습니다.

## 실무 가이드

- **핫패스**에서는 람다를 **값으로 전달**하고, 받는 쪽은 **템플릿**(또는 `std::invoke`와 함께)으로 받아 인라인을 유도합니다.
- **캡처 크기**가 크면(예: 큰 구조체), 람다가 그걸 값으로 캡처하면 클로저가 비대해지므로, **참조로 캡처**하되 람다 수명이 참조 대상보다 짧음을 보장하거나, **포인터/참조만** 캡처해 크기를 줄입니다.
- **람다 vs std::function**은 같은 로직으로 마이크로벤치마크를 돌려, 호출 비용·할당 유무 차이를 확인한 뒤, 핫 경로에서는 람다+템플릿 조합을 선택하는 것이 안전합니다.

## 평가 기준 (학습 성과 목표)

- **by-value** vs **by-reference** 캡처의 비용·수명 차이를 설명하고, 필요한 것만 명시적으로 캡처할 수 있다.
- 람다를 **템플릿 파라미터**로 받을 때 인라인 가능·**std::function**에 담으면 타입 소거·간접 호출·SBO 가능성을 구분할 수 있다.
- 핫패스에서 람다+템플릿 조합을 선택하고, 캡처 크기·수명을 고려해 설계할 수 있다.

## 판단 기준 (언제 쓸고 언제 피할지)

| 상황 | 권장 | 비권장 |
|------|------|--------|
| 콜백·알고리즘 전달 | 템플릿으로 F 받기 (인라인) | std::function에만 담기 |
| 작은 캡처 | 값 캡처 | [=]/[&]로 과다 캡처 |
| 큰 객체 참조 | 참조 캡처(수명 보장) | 값 캡처로 클로저 비대화 |

**적용 체크리스트**: (1) API는 템플릿으로 호출 가능 객체 받기. (2) 캡처는 필요한 것만 명시. (3) 람다 vs std::function 벤치마크.

## 비판적 시각: 한계와 트레이드오프

- **std::function**은 타입 소거가 필요할 때(저장·다형 콜백) 유용하다. "무조건 람다만"이 아니라, 저장·전달 경로가 핫하지 않으면 std::function도 선택지다.
- 참조 캡처 시 **dangling**을 피하려면 람다 수명이 참조 대상보다 짧음을 보장해야 한다.

### 자주 묻는 질문 (FAQ)

**Q: 람다 vs std::function 성능?**  
A: 람다(클로저)는 구체 타입이라 인라인 가능하고, std::function은 타입 소거로 간접 호출·SBO 여부에 따라 할당이 있을 수 있어 상대적으로 느립니다. 핫패스에는 템플릿으로 람다를 받는 것이 유리합니다.

**Q: by-value vs by-reference 캡처?**  
A: by-value는 복사 비용·클로저 크기 증가. by-reference는 수명 주의(뷰가 참조보다 오래 살지 않게). 작은 값은 value, 큰 값·수명 이슈 있으면 ref 또는 포인터/span.

**Q: std::function이 느린 이유는?**  
A: 타입 소거로 인라인되지 않고, 작은 객체는 SBO로 힙 할당을 피할 수 있으나(챕터 14) 호출은 간접입니다. 템플릿으로 구체 타입을 받으면 인라인됩니다.

### 자주 하는 실수

- **핫패스에 std::function만 사용**: 템플릿으로 람다를 받으면 인라인 가능; function은 간접 호출.
- **by-reference 캡처 후 수명 무시**: 람다가 참조보다 오래 살면 미정의 동작; 수명을 보장하거나 복사합니다.
- **큰 객체를 by-value 캡처**: 클로저 크기·복사 비용 증가; ref 또는 포인터/span 고려.

### 리팩토링 시 주의

std::function 인자를 템플릿으로 바꿀 때 호출처가 구체 타입을 유지하므로 인라인 기회가 생깁니다. 캡처를 바꿀 때 수명·크기를 재확인하고 벤치마크로 효과를 확인합니다.

### 실전 시나리오: 콜백 API

핫패스 콜백은 템플릿으로 받아 인라인을 유도하고, 저장이 필요할 때만 std::function 등 타입 소거를 사용합니다. 클로저가 크면 SBO를 벗어나 힙 할당이 발생할 수 있으므로(챕터 14) 크기를 의식합니다.

### 상황별 권장 (요약 표)

| 상황 | 권장 | 비권장 |
|------|------|--------|
| 핫패스 콜백 | 템플릿으로 람다 받기 | std::function만 사용 |
| 캡처 | 작은 값=value, 큰/수명 이슈=ref | 무분별한 value |
| 저장 필요 | std::function(SBO 한계 인지) | 템플릿만으로 저장 불가 시 |

### 컴파일러·플랫폼별 참고

- **클로저**: 익명 타입, 크기는 캡처에 따라 다름. **std::function**: 구현체별 SBO 크기 다름(챕터 14).
- **벤치마크**: 동일 로직으로 람다(템플릿) vs function 비교 권장.

- 람다·클로저와 std::function의 성능 차이를 설명할 수 있다.

### 정량적 비교 (람다 vs std::function)

| 측면 | 람다(템플릿으로 받음) | std::function |
|------|------------------------|---------------|

| 할당 | 없음(클로저 자체) | SBO 한도 내면 없음, 넘으면 힙 |
| 캡처 | value=복사·크기, ref=수명 주의 | 동일 |

### 실무 팁

- 핫패스 콜백은 템플릿으로 람다를 받아 인라인을 유도합니다.

- std::function 저장이 필요할 때는 클로저 크기가 SBO를 넘지 않게 의식합니다(챕터 14).
