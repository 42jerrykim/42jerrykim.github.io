**최적화 플래그**는 코드를 바꾸지 않고도 생성되는 기계어와 성능을 바꾸는 첫 단계입니다. 같은 소스에 **-O2**와 **-O3**만 바꿔도 벤치마크 결과가 수 퍼센트에서 수십 퍼센트까지 달라질 수 있으며, 잘못 선택하면 코드 크기 증가로 오히려 느려지는 **회귀**도 발생합니다. 이 챕터에서는 GCC/Clang의 **-O0**~**-Ofast**와 MSVC의 **/O1**·**/O2**·**/Ox** 의미, trade-off, **언제 어떤 레벨을 쓸지/피할지** 판단 기준, 그리고 릴리즈/디버그/프로파일 빌드 전략을 정리합니다.

## 왜 최적화 플래그인가 (동기)

컴파일러는 소스를 **중간 표현(IR)**으로 바꾼 뒤, 수십 가지 최적화 패스(인라이닝, 루프 최적화, 레지스터 할당, 벡터화 등)를 적용해 기계어를 만듭니다. 어떤 패스를 켤지는 **최적화 레벨** 한 번으로 제어할 수 있어, 코드 수정 없이 빌드 옵션만으로 성능을 탐색할 수 있습니다.

## 역사·배경 (최적화 레벨의 등장)

GCC에서는 1990년대부터 **-O**, **-O2** 등 최적화 레벨이 도입되었고, **-O3**는 더 공격적인 최적화를 묶은 것으로 확장되었습니다. GCC 공식 문서에서는 -O2에 대해 "대부분의 최적화를 수행한다"고 설명합니다.

> "Optimize even more. -O3 turns on all optimizations specified by -O2 and also turns on the -finline-functions, -funswitch-loops, -fpredictive-commoning, -fgcse-after-reload, -ftree-vectorize and other options." — GCC Manual, Optimize Options

위와 같이 레벨이 올라갈수록 추가로 켜지는 옵션이 문서에 나열되어 있으므로, 정확한 목록은 사용 중인 GCC 버전의 문서를 참고하는 것이 좋습니다. **-Os**(크기 최소화)는 임베디드·제한된 환경을 위해 추가되었고, **-Ofast**는 GCC 4.x 대에서 부동소수점 완화 옵션을 묶은 레벨로 등장했습니다. Clang은 GCC와 호환되는 레벨 이름을 유지하면서 내부적으로 다른 최적화 순서·휴리스틱을 쓰며, MSVC는 **/O1**, **/O2**, **/Ox** 등 자체 명칭을 사용합니다. 오늘날 대부분의 C/C++ 프로젝트는 **-O2** 또는 **/O2**를 릴리즈 기본값으로 두고, 필요 시 **-O3**를 벤치마크와 함께 검증하는 패턴이 널리 쓰입니다. **최적화 레벨**은 "어떤 패스를 얼마나 공격적으로 켤지"를 한 번에 지정하는 옵션입니다. 레벨을 올리면 더 많은 최적화가 켜지지만, 컴파일 시간·메모리·코드 크기가 늘고, 드물게 특정 코드에서 성능이 나빠지는 경우도 있습니다. 따라서 **측정 후 선택**하는 것이 중요하며, 이 챕터는 그 선택을 위한 정의·예시·판단 기준을 제공합니다.

## 최적화 레벨 정의 (GCC / Clang)

GCC와 Clang은 다음 최적화 레벨 옵션을 제공합니다. 각 레벨에서 **켜지는 최적화 종류**와 **목적**이 다릅니다.

- **-O0**: **최적화 없음**. 디버깅 시 변수·소스 라인 대응이 정확하고 컴파일이 빠르지만, 생성 코드는 느립니다. 디버그 빌드 기본값으로 적합합니다.
- **-O1**: **기본적인 최적화**만 적용. 컴파일 시간을 크게 늘리지 않으면서 일부 성능 이득을 얻을 때 사용합니다. 인라인은 제한적, 루프·레지스터 최적화가 일부 켜집니다.
- **-O2**: **대부분의 릴리즈 빌드 권장 레벨**입니다. 인라이닝, 루프 최적화, 레지스터 할당 등 대다수 최적화가 켜지고, -O3에 비해 공격적인 변환은 적어 코드 크기·컴파일 시간·회귀 위험이 상대적으로 낮습니다.
- **-O3**: -O2에 더해 **더 공격적인 인라이닝**, **루프 언롤링**, 예측 실행(prediction) 관련 최적화가 추가됩니다. 수치 연산·벡터화에 유리한 코드에서는 -O2 대비 눈에 띄게 빨라질 수 있지만, 코드 크기가 커지고 컴파일 시간·메모리 사용량이 늘며, 드물게 **성능 회귀**가 나오는 경우도 있습니다.
- **-Os**: 코드 **크기**를 줄이는 방향으로 최적화합니다. 인라이닝·언롤링을 억제해 바이너리 크기를 줄이므로, 임베디드·캐시가 극히 제한된 환경에서 고려합니다.
- **-Ofast**: -O3에 더해 **부동소수점 연산의 표준 준수**를 완화합니다(예: 연관 법칙 가정, -ffast-math 계열). 수치적으로 더 빠른 코드가 나올 수 있지만, 반올림·연산 순서에 따라 결과가 달라질 수 있어, 신호 처리·과학 계산이 아닌 일반 코드에서는 사용을 피하는 것이 안전합니다.

### MSVC 대응

MSVC는 **/O1**(크기 최소화), **/O2**(속도 최대화, 기본 릴리즈), **/Ox**(최대 최적화) 등으로 비슷한 역할을 합니다. /O2가 GCC/Clang의 -O2에 대응하는 기본 릴리즈 설정으로 널리 쓰입니다. Visual Studio 프로젝트 속성에서 "Optimization"을 "Maximize Speed (/O2)"로 두면 됩니다.

## 최적화 레벨별 켜지는 최적화 (개념)

레벨이 올라갈수록 인라이닝·루프 최적화·벡터화 등이 더 공격적으로 적용됩니다. 개념적으로는 아래와 같습니다.

```mermaid
flowchart LR
  subgraph low ["-O0 / -O1"]
    A["최소 최적화"]
    B["디버깅 용이"]
  end
  subgraph mid ["-O2"]
    C["인라이닝·루프·레지스터"]
    D["일반 릴리즈 권장"]
  end
  subgraph high ["-O3 / -Ofast"]
    E["공격적 인라이닝·언롤링"]
    F["코드 크기·회귀 주의"]
  end
  low --> mid --> high
```

(실제로 켜지는 플래그 목록은 컴파일러·버전마다 다르므로, `gcc -O2 -Q --help=optimizers` 또는 문서를 참고하는 것이 좋습니다.)

## -O2 vs -O3 trade-off

- **속도**: 벡터화·인라이닝에 민감한 핫 루프가 있으면 -O3가 유리한 경우가 많습니다. 차이는 워크로드에 따라 0%에 가깝게 나올 수도 있고, 두 자릿수 %까지 나올 수도 있습니다.
- **코드 크기**: -O3는 인라이닝·언롤링으로 텍스트 세그먼트가 커져 **I-cache** 압박이 생길 수 있어, 작은 핫 루프가 많은 경우 오히려 느려질 수 있습니다.
- **컴파일 시간**: -O3가 더 오래 걸리고, 대형 프로젝트에서는 빌드 시간이 눈에 띄게 늘어납니다.
- **회귀**: -O3에서만 발생하는 회귀는 드물지만 보고되는 편이므로, 릴리즈 전에 **동일 벤치마크로 -O2 vs -O3**를 비교해 두는 것이 좋습니다.

## -Ofast 위험 (부동소수점)

**-Ofast**는 **-ffast-math** 계열 옵션을 켜서 부동소수점 연산 순서·정밀도를 바꿉니다. 금융·규제·재현성이 중요한 코드에서는 사용하지 않고, 필요 시 -O3 + 개별 fast-math 옵션만 선택적으로 켜는 편이 안전합니다.

## 예시: 빌드 명령과 크기·시간 비교

동일한 소스에 다른 최적화 레벨을 적용해 컴파일 시간과 바이너리 크기를 비교하는 예시입니다.

```bash
# 단일 TU 컴파일 시간 비교 (GCC/Clang)
g++ -O0 -c example.cc -o example_o0.o   # 가장 빠름
g++ -O2 -c example.cc -o example_o2.o
g++ -O3 -c example.cc -o example_o3.o # 가장 느림

# 실행 파일 크기 비교 (텍스트 세그먼트)
g++ -O2 example.cc -o app_o2 && size app_o2
g++ -O3 example.cc -o app_o3 && size app_o3
# -O3 쪽 text 영역이 더 클 수 있음
```

```bash
# CMake에서 Release/Debug 설정
cmake -DCMAKE_BUILD_TYPE=Release ..   # 보통 -O3 또는 -O2
cmake -DCMAKE_BUILD_TYPE=Debug ..     # -O0 -g
cmake -DCMAKE_BUILD_TYPE=RelWithDebInfo ..  # -O2 -g
```

실제 벤치마크는 **동일한 벤치마크 스위트**를 각 플래그로 빌드한 뒤 반복 실행해 평균·분포를 비교합니다. 벤치마크가 핫패스를 잘 커버해야 의미 있는 차이가 나옵니다.

## 측정과 비교 (실무 체크리스트)

- **컴파일 시간**: 동일 소스에 -O0 / -O2 / -O3를 적용해 컴파일 시간을 측정합니다. 대형 TU에서는 차이가 크게 납니다.
- **실행 파일 크기**: `size` 또는 `ls -l`로 텍스트·데이터·bss 크기를 비교합니다. -Os vs -O2 vs -O3 순으로 크기가 달라집니다.
- **벤치마크**: 동일한 벤치마크 스위트를 각 플래그로 빌드한 뒤 반복 실행해 평균·분포를 비교합니다. 벤치마크가 핫패스를 잘 커버해야 의미 있는 차이가 나옵니다.
- **회귀 테스트**: 릴리즈 브랜치에서는 -O2 또는 -O3를 고정하고, 코드 변경 시 동일 벤치마크로 성능이 떨어지지 않는지 CI에서 주기적으로 확인하는 것을 권장합니다.

## 릴리즈/디버그/프로파일 빌드 전략

- **디버그**: -O0 (또는 -O1). 디버그 정보(`-g`)와 함께 사용해 단계 실행·변수 조사에 적합하게 유지합니다.
- **릴리즈**: -O2를 기본으로 하고, 핫 경로가 수치·벡터화에 민감하면 -O3를 시도한 뒤 **벤치마크로 검증**합니다. 회귀가 없을 때만 -O3를 채택합니다.
- **프로파일 수집**: PGO 1단계처럼 instrumented 빌드에서는 -O2 수준으로 두고, 프로파일 기반 최적화 단계에서 -O3를 쓰는 조합이 흔합니다.

빌드 시스템(CMake, Meson 등)에서 **Release**, **RelWithDebInfo**, **Debug** 등으로 구분해 두고, CI에서는 Release(-O2 또는 -O3)로 성능 회귀를 주기적으로 측정하는 것을 권장합니다.

## 한눈에 보기: 최적화 레벨 비교

| 레벨 | 목적 | 속도 | 코드 크기 | 컴파일 시간 | 권장 용도 |
|------|------|------|-----------|-------------|-----------|
| -O0 | 없음 | 느림 | 작음 | 빠름 | 디버그 빌드 |
| -O1 | 기본 | 보통 | 보통 | 보통 | 빠른 빌드 + 일부 이득 |
| -O2 | 균형 | 빠름 | 보통 | 보통 | **일반 릴리즈 기본** |
| -O3 | 최대 속도 | 매우 빠를 수 있음 | 큼 | 느림 | 수치/벡터화 핫패스, 벤치 검증 후 |
| -Os | 크기 최소 | 보통 | 작음 | 보통 | 임베디드·제한된 캐시 |
| -Ofast | 수치 최대 | 부동소수점 더 빠를 수 있음 | 큼 | 느림 | 신호/과학 계산만, 재현성 불필요 시 |

## 판단 기준: 언제 쓸지 / 언제 피할지

| 상황 | 권장 | 비권장 |
|------|------|--------|
| 일반 릴리즈 빌드 | -O2 | -O0, -Ofast(일반 코드) |
| 수치·벡터화가 중요한 핫 루프 | -O3 (벤치로 검증 후) | -O3 무검증 채택 |
| 디버깅·단계 실행 | -O0 또는 -O1 | -O2 이상 |
| 임베디드·바이너리 크기 제한 | -Os 또는 -O2 | -O3 |
| 금융·규제·재현성 중요 | -O2 또는 -O3 (fast-math 없음) | -Ofast |
| PGO 1단계(instrumented) | -O2 수준 | -O0(너무 느림), -O3(과도) |

**적용 체크리스트**: (1) 릴리즈 기본은 -O2로 두고, -O3는 벤치마크로 이득과 회귀를 확인한 뒤 도입한다. (2) -Ofast는 부동소수점 재현성이 필요 없는 도메인에서만 사용한다. (3) CI에서 릴리즈 빌드로 성능 회귀를 주기적으로 측정한다.

## 자주 하는 실수

- **-O3를 검증 없이 릴리즈에 넣는 경우**: 일부 코드에서 코드 크기 증가로 느려질 수 있으므로, 동일 벤치마크로 -O2 vs -O3를 비교한 뒤 채택한다.
- **-Ofast를 일반 비즈니스 로직에 쓰는 경우**: 부동소수점 결과가 바뀔 수 있어, 재현성·규제가 중요한 코드에서는 사용하지 않는다.
- **디버그 빌드에 -O2 이상을 쓰는 경우**: 변수 최적화로 사라지거나 라인 대응이 어긋나 디버깅이 어려워질 수 있으므로, 디버깅 시에는 -O0 또는 -O1을 유지한다.

## 비판적 시각: 한계와 트레이드오프

최적화 레벨은 **높을수록 좋다**가 아닙니다. -O3는 코드 크기를 키워 I-cache 미스를 늘릴 수 있고, -Ofast는 수치 결과를 바꿉니다. "릴리즈 = -O3"로 고정하기보다, **프로젝트의 핫패스와 벤치마크로 한 번씩 측정**한 뒤 선택하는 것이 안전합니다. 레거시 프로젝트에서는 기존에 -O2로 잘 돌아가고 있다면, -O3 전환은 성능 이득이 명확할 때만 단계적으로 도입하는 것이 좋습니다.

- **-O2 vs -O3** trade-off(속도·코드 크기·컴파일 시간·회귀)를 설명하고, 상황에 맞게 선택할 수 있다.

## Course 01(인라이닝)과의 연계

Low-latency C++ Language Optimization 트랙(Course 01)에서는 **인라이닝 유도**를 다룹니다. 인라이닝이 되려면 컴파일러가 호출 지점에서 함수 정의를 볼 수 있어야 하고, **최적화 레벨**이 -O1 이상이어야 합니다. -O0에서는 인라이닝이 거의 켜지지 않습니다. 따라서 "인라이닝을 유도하는 코드 배치·final·LTO"는 **-O2 또는 -O3**와 함께 쓸 때 효과가 납니다. 이 챕터는 "어떤 레벨을 켤지"를 다루고, Course 01은 "그 레벨에서 인라이닝이 실제로 일어나도록 코드를 어떻게 쓸지"를 다룹니다.

## 예시: 벤치마크 결과 해석

가정: 동일 벤치마크를 -O2 빌드와 -O3 빌드로 각각 10회 실행해 평균 지연 시간을 수집했습니다. 결과가 -O2: 100µs, -O3: 85µs 라면 약 15% 개선입니다. 이때 (1) 표준편차나 백분위도 확인해 차이가 통계적으로 의미 있는지 봅니다. (2) 다른 벤치마크(같은 스위트 내 다른 케이스)에서 -O3가 느려진 구간이 없는지 확인합니다. (3) -O3 바이너리의 텍스트 세그먼트 크기가 -O2 대비 크게 늘었는지 확인합니다. 모든 구간에서 이득이 있거나 동등하고, 크기 증가가 수용 가능하면 -O3를 릴리즈에 채택할 수 있습니다. 한 구간이라도 -O3에서 나빠지면 해당 구간 원인(예: I-cache 압박)을 분석하거나, 해당 모듈만 -O2로 두는 방안을 검토합니다.

## 용어 정리

| 용어 | 설명 |
|------|------|
| **-O0~-O3** | GCC/Clang 최적화 레벨; 숫자가 클수록 더 많은·공격적인 최적화 적용 |
| **-Os** | 크기 최소화 방향 최적화; 인라이닝·언롤링 억제 |
| **-Ofast** | -O3 + 부동소수점 표준 완화(-ffast-math 계열) |
| **IR** | Intermediate Representation; 컴파일러 내부 중간 표현 |
| **I-cache** | Instruction cache; 명령어 캐시; 코드 크기가 크면 미스 증가 가능 |
| **회귀** | 최적화 레벨을 올렸을 때 특정 코드에서 성능이 나빠지는 현상 |
| **RelWithDebInfo** | Release with Debug Info; -O2 수준 + 디버그 심볼 |

## 실전 시나리오: -O2에서 -O3 전환 검토

1. **현재 상태**: 릴리즈 빌드가 -O2로 설정되어 있고, 핫패스에 수치 연산·루프가 많다고 가정합니다.
2. **벤치마크 준비**: 해당 핫패스를 잘 커버하는 벤치마크(또는 통합 테스트의 성능 구간)를 확보합니다.
3. **빌드**: 동일 소스로 -O2와 -O3 각각 빌드해 두 개의 바이너리를 만듭니다.
4. **실행·측정**: 같은 벤치마크를 여러 번 실행해 평균·표준편차(또는 백분위)를 기록합니다.
5. **크기 확인**: `size`로 텍스트 세그먼트가 얼마나 늘었는지 확인합니다. 크기가 크게 늘고 지연이 중요한 경로가 여러 개면 I-cache 압박 가능성을 염두에 둡니다.
6. **판단**: -O3에서 속도 이득이 명확하고 회귀(다른 구간이 느려지는 경우)가 없으면 릴리즈를 -O3로 전환합니다. 이득이 미미하거나 회귀가 있으면 -O2를 유지합니다.
7. **CI 반영**: 선택한 레벨(-O2 또는 -O3)로 릴리즈 빌드 후 동일 벤치마크를 CI에서 주기적으로 돌려 회귀를 감지합니다.

## 예시: 간단한 C++ 코드로 레벨별 차이 느끼기

아래는 작은 루프를 돌리는 예시입니다. -O0와 -O2로 빌드해 실행 시간 차이를 확인할 수 있습니다.

```cpp
// sum.cc: 단순 합산 루프 (벤치마크용 최소 예시)
#include <vector>
#include <cstdint>

uint64_t sum(const std::vector<int>& v) {
  uint64_t s = 0;
  for (size_t i = 0; i < v.size(); ++i)
    s += v[i];
  return s;
}
```

- **-O0**: 루프가 최소 최적화만 적용되어 상대적으로 느립니다.
- **-O2**: 루프 최적화·인라이닝으로 같은 소스가 훨씬 빠른 기계어로 생성됩니다.

실제 측정 시에는 Google Benchmark, nanobench 등으로 반복 실행 후 평균·분포를 보는 것이 좋습니다. 대형 TU에서는 `time g++ -O0 -c big.cc` vs `time g++ -O2 -c big.cc`로 컴파일 시간 차이도 확인할 수 있습니다.

## 고급: 개별 최적화 플래그와의 관계

최적화 레벨은 수십 개의 개별 플래그를 묶은 것입니다. 예를 들어 -O2는 대략 **-finline**, **-foptimize-sibling-calls**, **-fstrength-reduce** 등 수많은 **-f*** 옵션을 켭니다. -O3는 여기에 **-finline-functions**(더 공격적 인라이닝), **-funswitch-loops**, **-fprefetch-loop-arrays** 등이 더해진 묶음으로 이해할 수 있습니다. 실무에서는 보통 **-O2** 또는 **-O3**만 지정하고 개별 플래그를 건드리지 않는 것이 유지보수에 유리합니다. 다만 특정 버그가 특정 최적화에서만 발생한다고 알려진 경우(예: 루프 최적화로 인한 잘못된 결과), 해당 최적화만 끄는 **-fno-*** 플래그를 추가해 회귀를 피할 수 있습니다. 이때는 컴파일러 버전별로 어떤 플래그가 -O2/-O3에 포함되는지 문서를 확인하는 것이 좋습니다.

## 컴파일러별 최적화 옵션 확인 방법

어떤 최적화가 실제로 켜지는지 확인하고 싶을 때는 컴파일러 도움말을 사용할 수 있습니다.

- **GCC**: `gcc -O2 -Q --help=optimizers` 로 -O2에서 켜지는/꺼지는 최적화 목록을 볼 수 있습니다. `-O3 -Q --help=optimizers`로 -O3와의 차이도 비교할 수 있습니다.
- **Clang**: `clang -O2 -mllvm -debug-pass=Structure` (또는 버전에 따라 다른 디버그 출력)로 패스 구성을 볼 수 있으며, 공식 문서에서 -O2/-O3 차이를 확인할 수 있습니다.
- **MSVC**: 프로젝트 속성 "Optimization" 하위에서 "Inline Function Expansion", "Enable Intrinsic Functions" 등 개별 옵션을 볼 수 있습니다.

레벨만 바꾸고 개별 최적화를 세밀하게 제어하고 싶다면, -O2를 기본으로 두고 -fno-inline 같은 플래그로 특정 최적화만 끄는 방식도 가능합니다. 다만 대부분의 경우 **-O2 vs -O3** 두 가지를 벤치로 비교하는 것만으로도 실무에서 충분한 판단이 가능합니다.

## 빌드 시스템별 설정 요약

| 빌드 시스템 | 릴리즈(-O2) | 디버그(-O0) | RelWithDebInfo |
|-------------|-------------|-------------|----------------|
| CMake | CMAKE_BUILD_TYPE=Release | CMAKE_BUILD_TYPE=Debug | CMAKE_BUILD_TYPE=RelWithDebInfo |
| Meson | buildtype: 'release' | buildtype: 'debug' | buildtype: 'debugoptimized' |
| Make | CXXFLAGS=-O2 | CXXFLAGS=-O0 -g | CXXFLAGS=-O2 -g |

CMake에서는 Release 시 기본이 보통 -O3인 경우도 있으므로, 프로젝트에서 `add_compile_options(-O2)` 등으로 명시적으로 고정해 두면 환경에 덜 흔들립니다.

## 시나리오 2: 디버그 빌드에서 변수가 최적화로 사라짐

증상: -O2 또는 -O3로 빌드한 뒤 디버거로 변수를 보려 하면 "optimized out"으로 나오거나 값이 보이지 않습니다. 원인: 컴파일러가 사용되지 않는다고 판단한 변수를 제거하거나 레지스터에만 두어, 디버그 정보와 소스 대응이 어긋납니다. 대처: 해당 빌드에서는 **-O0** 또는 **-O1**을 쓰고 **-g**를 켜서 디버그 빌드를 유지합니다. 릴리즈와 동일한 최적화로 재현하고 싶다면 **RelWithDebInfo**(-O2 -g)를 쓰되, 일부 변수는 여전히 최적화되어 보이지 않을 수 있음을 받아들입니다. 가능하면 문제 재현용 최소 재현 케이스를 -O0로 만들어 버그 위치를 좁힌 뒤, 릴리즈 빌드에서의 동작은 별도로 검증합니다.

## 시나리오 3: CI에서 -O2 vs -O3 자동 비교

목표: 코드 변경이 올라올 때마다 -O2와 -O3 빌드 각각으로 벤치마크를 돌려, -O3에서만 느려지는 회귀를 자동으로 잡고 싶습니다. 절차: (1) CI 스크립트에서 동일 소스로 두 번 빌드합니다. 예: `cmake -DCMAKE_BUILD_TYPE=Release -DEXTRA_CXX_FLAGS=-O2` 와 `-O3` 각각. (2) 동일한 벤치마크 타깃을 두 빌드에서 실행하고, 결과(평균 지연 또는 throughput)를 수집합니다. (3) 기준선(이전 커밋 또는 main 브랜치)과 비교하거나, -O2 대비 -O3 결과가 일정 비율 이상 나빠지면 실패로 처리합니다. (4) 실패 시 아티팩트(빌드 로그·벤치 결과)를 보관해 원인 분석에 사용합니다. 이렇게 하면 -O3를 릴리즈에 쓰는 프로젝트에서 회귀를 줄일 수 있습니다.

## 부동소수점과 -Ofast 상세

-Ofast는 **-ffast-math**를 켜며, 이는 여러 하위 옵션의 묶음입니다. 예: **-fno-math-errno**, **-ffinite-math-only**, **-fno-rounding-math**, **-fno-signaling-nans**, **-fassociative-math** 등. **연관 법칙**을 가정하면 (a + b) + c와 a + (b + c)를 같은 것으로 간주해 순서를 바꿀 수 있어, 벡터화 등이 더 잘 들어갈 수 있지만, 부동소수점에서는 순서에 따라 결과가 달라질 수 있습니다. 따라서 재현성이 중요한 시뮬레이션·금융·규제 제출 코드에서는 -Ofast를 쓰지 않고, -O3까지로 제한합니다. 신호 처리·이미지 처리처럼 오차 허용 범위가 넓은 도메인에서는 -Ofast로 속도 이득을 취하는 경우가 있습니다. 필요 시 -O3에 **-ffast-math** 대신 **-fno-math-errno** 같은 개별 옵션만 켜서 위험을 줄일 수 있습니다.

## 자주 묻는 질문

- **Q: 릴리즈는 무조건 -O3가 맞나요?** A: 아닙니다. -O2를 기본으로 하고, 벤치마크로 -O3 이득이 있고 회귀가 없을 때만 -O3를 씁니다.
- **Q: -Os와 -O2 중 무엇을 써야 하나요?** A: 바이너리 크기나 I-cache가 극히 제한된 환경(일부 임베디드)이면 -Os를 고려하고, 그렇지 않으면 보통 -O2가 유리합니다.
- **Q: 디버그 빌드에 -O2를 써도 되나요?** A: 단계 실행·변수 조사가 어긋날 수 있으므로, 순수 디버깅 목적이면 -O0 또는 -O1을 권장합니다.
- **Q: -Ofast를 썼을 때 테스트가 실패합니다.** A: 부동소수점 비교(기대값 vs 실제값)가 비트 단위로 맞지 않을 수 있습니다. 테스트에서 허용 오차(epsilon)를 두거나, 해당 모듈만 -O3로 두고 -Ofast를 제거합니다.
- **Q: CMake Release가 -O3인데 -O2로 고정하려면?** A: `set(CMAKE_CXX_FLAGS_RELEASE "-O2 -DNDEBUG")` 처럼 프로젝트에서 Release 플래그를 덮어쓸 수 있습니다.
- **Q: PGO와 최적화 레벨 관계는?** A: PGO 1단계(instrumented 빌드)는 보통 -O2 수준으로 두고, 2단계(프로파일 반영 빌드)에서 -O3를 쓰는 조합이 많습니다. 챕터 03에서 다룹니다.
- **Q: 컴파일 시간을 줄이려면?** A: -O3 대신 -O2를 쓰면 줄어듭니다. 대형 TU를 나누거나, ccache/sccache로 재컴파일을 줄이는 방법은 챕터 12에서 다룹니다.

|------|----------|-------------|--------|-----------|-------------|---------|
| -O0 | 거의 없음 | 없음 | 없음 | 최소 | 최단 | 디버깅 |
| -O1 | 제한적 | 일부 | 제한적 | 작음 | 짧음 | 빠른 빌드+일부 이득 |
| -O2 | 적극적 | 대부분 | 일반적 | 보통 | 보통 | **일반 릴리즈** |
| -O3 | 매우 적극적 | 언롤링 등 추가 | 적극적 | 큼 | 김 | 수치/벡터화 핫패스 |
| -Os | 억제 | 보통 | 제한적 | 최소화 | 보통 | 임베디드/크기 제한 |
| -Ofast | -O3와 동일 | -O3와 동일 | -O3+추가 | 큼 | 김 | 부동소수점 전용(주의) |

위 표는 개념적 요약이며, 실제로 켜지는 옵션은 컴파일러·버전에 따라 다릅니다.

## LTO·PGO와의 조합

**LTO**(챕터 02)를 쓸 때도 각 번역 단위는 먼저 -O2 또는 -O3로 컴파일된 IR을 생성하고, 링크 시점에 그 IR을 합쳐 추가 최적화를 합니다. 따라서 **-O2 -flto** 또는 **-O3 -flto** 조합이 일반적입니다. **PGO**(챕터 03)에서는 1단계(instrumented 빌드)를 보통 -O2로 하고, 2단계(프로파일 반영)에서 -O3를 쓰는 패턴이 많습니다. 최적화 레벨을 올리는 것만으로도 이득이 있고, LTO·PGO와 함께 쓰면 추가 이득이 나올 수 있으나, 항상 **동일 벤치마크로 측정**해 회귀가 없는지 확인하는 것이 좋습니다.

## 실무 권장 요약

- **새 프로젝트**: 릴리즈는 **-O2**를 기본으로 두고, 핫패스가 수치·벡터화에 민감하면 **-O3**를 벤치마크로 검증한 뒤 선택적으로 켭니다.
- **기존 프로젝트**: 이미 -O2로 안정적이면 무리하게 -O3로 올리지 않고, 성능 목표가 있을 때만 -O2 vs -O3를 측정해 전환합니다.
- **디버그**: 항상 **-O0** 또는 **-O1**과 **-g**를 유지해, 변수·소스 대응이 깨지지 않도록 합니다.
- **-Ofast**: 부동소수점 재현성이 필요 없는 도메인(신호 처리·일부 과학 계산)에서만 사용하고, 금융·규제·테스트 재현성은 **-O3**까지로 제한합니다.
- **CI**: 릴리즈 빌드(-O2 또는 -O3)로 성능 회귀 테스트를 주기적으로 돌리고, 가능하면 -O2와 -O3 결과를 둘 다 수집해 비교합니다.

## 리팩토링 시 주의

기존에 -O2로 릴리즈하던 프로젝트를 -O3로 바꿀 때는 **한 번에 전체 전환**하기보다, 먼저 벤치마크로 이득과 회귀를 확인한 뒤 전환하는 것이 안전합니다. 이미 -O3를 쓰고 있는데 특정 TU만 문제를 일으킨다면, 해당 TU만 -O2로 컴파일하도록 예외를 두는 방법도 있습니다(CMake: `set_source_files_properties(... COMPILE_FLAGS -O2)`). -Ofast는 새로 도입할 때 특히 신중해야 하며, 부동소수점을 쓰는 모듈이 있다면 해당 모듈만 -O3로 두고 나머지는 -O2로 두는 식으로 격리할 수 있습니다.

## 이 장에서 다룬 내용

- **최적화 플래그**의 동기: 코드 변경 없이 생성 기계어와 성능을 바꾸는 첫 단계.
- **정의**: -O0, -O1, -O2, -O3, -Os, -Ofast(GCC/Clang)와 /O1, /O2, /Ox(MSVC)의 의미와 켜지는 최적화 방향.
- **-O2 vs -O3** trade-off: 속도·코드 크기·컴파일 시간·회귀; 벤치마크로 검증 후 선택.
- **-Ofast** 위험: 부동소수점 재현성; 금융·규제 코드에서는 사용 금지.
- **측정과 비교**: 컴파일 시간·실행 파일 크기·벤치마크; CI에서 회귀 감지.
- **릴리즈/디버그/프로파일** 빌드 전략과 적용 체크리스트.
- **판단 기준**: 언제 -O2/-O3/-Ofast를 쓸지·피할지 표와 체크리스트.
- **비판적 시각**: 최적화 레벨은 높을수록 좋은 것이 아니며, 측정 후 선택하는 습관이 필요함.
