## 왜 소프트웨어 아키텍처를 배우는가

기능 하나를 추가하는 데 반나절이면 충분했던 프로젝트가, 3년이 지나면 같은 규모의 변경에 몇 주가 걸리는 경우를 흔히 본다. 코드 품질이 갑자기 나빠져서가 아니라, 모듈 간 경계와 의존성 방향을 관리하는 **아키텍처**가 없었기 때문이다. 소프트웨어 아키텍처는 시스템을 구성하는 요소들과 그 관계, 그리고 시스템의 진화를 이끄는 원칙의 집합이다. 기능 명세가 "무엇을 만들 것인가"를 답한다면, 아키텍처는 "그 기능들이 서로 어떻게 맞물려야 변경 비용이 시간에 비례해 폭발하지 않는가"를 답한다.

이 교육과정을 배워야 하는 이유는 세 가지로 요약된다. 첫째, 주니어 개발자에서 시니어로 성장하는 과정에서 가장 먼저 요구받는 역량이 "이 기능을 어느 모듈에, 어떤 경계로 넣을 것인가"를 판단하는 능력이며, 이는 개별 패턴 암기가 아니라 아키텍처 원칙의 체화로만 얻어진다. 둘째, 마이크로서비스·이벤트 기반 아키텍처·클라우드 네이티브처럼 현대 시스템을 설명하는 용어 대부분이 특정 회사의 유행어가 아니라 분산 시스템 이론과 도메인 주도 설계라는 공통 기반 위에 서 있어서, 기반 없이 용어만 익히면 응용 상황에서 오판하기 쉽다. 셋째, 아키텍처 결정은 되돌리기 가장 비싼 결정이므로, 그 결정을 내리는 사람(아키텍트뿐 아니라 팀 리드·시니어 개발자)이 트레이드오프를 언어화하지 못하면 조직 전체가 잘못된 결정의 비용을 반복해서 치르게 된다.

## 교육과정 구성과 학습 경로

이 시리즈는 기본과정(1~8장)과 고급과정(9~16장)의 두 단계로 구성된다. 기본과정은 아키텍처의 정의·원칙·패턴·품질 속성·문서화·평가·실무 사례를 다루며, 프로그래밍 경험은 있지만 시스템 전체를 설계해 본 적은 없는 주니어~미드레벨 개발자를 대상으로 한다. 고급과정은 도메인 주도 설계(DDD), 데이터 아키텍처, 분산 시스템, 클라우드 네이티브, API 통합, 엔터프라이즈 거버넌스, 고급 실무를 다루며, 기본과정을 마쳤거나 이에 준하는 실무 경험이 있는 시니어 개발자·아키텍트를 대상으로 한다. 두 과정을 가르는 기준은 난이도가 아니라 **의사결정의 범위**다 — 기본과정은 "이 모듈을 어떻게 나눌 것인가"를, 고급과정은 "여러 팀·여러 서비스·여러 데이터센터에 걸친 시스템을 어떻게 진화시킬 것인가"를 다룬다.

챕터 순서는 임의가 아니라 의존관계를 따른다. 1~3장에서 아키텍처의 정의, SOLID를 비롯한 설계 원칙, 계층화·이벤트 기반 같은 고전 패턴을 먼저 다지지 않으면 4장의 마이크로서비스·헥사고날 아키텍처가 왜 그런 형태를 갖는지 이해할 수 없다. 5~6장의 품질 속성과 문서화는 7~8장의 평가·실무 사례를 읽을 때 "무엇을 기준으로 좋다/나쁘다를 판단하는가"의 어휘를 제공한다. 고급과정에서는 9~10장의 DDD가 11장 데이터 아키텍처와 12장 분산 시스템의 전제가 되는데, 바운디드 컨텍스트 없이 데이터를 나누면 어디서 자르든 임의적이기 때문이다. 13~14장의 클라우드 네이티브·API 통합은 12장의 분산 시스템 이론을 실제 인프라 위에 구현하는 단계이고, 15~16장의 거버넌스·고급 실무는 그렇게 만든 시스템을 조직 차원에서 지속 가능하게 운영하는 방법을 다룬다. 아래 표는 각 챕터가 속한 단계와 핵심 주제를 정리한 것이다.

## 완주 후 갖추게 되는 역량

기본과정을 마치면 아키텍처와 상세 설계의 경계를 구분하고, 계층화·클라이언트-서버·이벤트 기반 같은 고전 패턴과 마이크로서비스·헥사고날 같은 현대 패턴 중 상황에 맞는 것을 근거를 들어 선택할 수 있게 된다. 성능·확장성·가용성 같은 품질 속성을 요구사항 단계에서 식별해 아키텍처 결정에 반영하고, C4 모델 등으로 그 결정을 팀에 문서화·전달하는 능력도 포함된다. 고급과정을 마치면 복잡한 도메인을 바운디드 컨텍스트로 분해하고 전술적 DDD 패턴(엔티티·값 객체·애그리게이트)으로 구현하며, CAP 정리와 Saga 패턴 같은 분산 시스템 이론을 근거로 데이터 일관성 전략을 설계할 수 있게 된다. 여기에 클라우드 네이티브 인프라 위에서 API 게이트웨이·서비스 메시로 시스템을 통합하고, 조직 차원의 아키텍처 거버넌스를 설계하는 역량까지 더해지면, 개별 시스템 설계자를 넘어 여러 팀의 기술 방향을 조율하는 아키텍트 역할을 맡을 준비가 된다.

각 챕터는 핵심 개념을 문단으로 설명한 뒤 비교표·다이어그램·코드로 보충하는 순서를 따르며, 장 말미에는 평가 기준과 다음 장으로의 연결을 둔다. 기본과정만으로도 실무에서 아키텍처 결정에 참여할 최소 역량을 얻을 수 있으므로, 시니어 개발자 이상이 아니라면 [1장 소프트웨어 아키텍처 기초](/post/software-architecture/software-architecture-fundamentals/)부터 순서대로 읽는 것을 권장한다. 이미 마이크로서비스·클라우드 실무 경험이 있는 독자는 [9장 DDD 기초: 전략적 설계](/post/software-architecture/ddd-strategic-design-fundamentals/)부터 시작해도 무리가 없지만, 그 경우에도 [2장(설계 원칙)](/post/software-architecture/architecture-design-principles/)과 [4장(모던 아키텍처 패러다임)](/post/software-architecture/modern-architecture-paradigms/)은 이후 장에서 반복적으로 참조되므로 먼저 훑어보는 편이 낫다.
